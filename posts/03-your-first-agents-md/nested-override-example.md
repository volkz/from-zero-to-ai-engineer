# Folder-Level AGENTS.md Override Example

*From 0 to AI Engineer — Post 03 artifact*

This is a worked example of how a nested AGENTS.md narrows and extends the root manifest for a specific package in a monorepo — without duplicating the global rules.

---

## Scenario

Root `AGENTS.md` defines global rules for the whole repo:
- Package manager: yarn
- Test command: `yarn test`
- Global Nest patterns apply everywhere

But the `platform/bff/` package (a BFF — Backend For Frontend) has different needs:
- Different test command (needs `--runInBand` flag to avoid race conditions)
- BFF-specific HTTP patterns the root doesn't know about
- Different folder ownership rules

---

## Root AGENTS.md (excerpt)

```markdown
## Package manager
yarn — never use npm or pnpm

## Exact commands
- Test: yarn test
- Lint: yarn lint

## Forbidden
- Never use npm
- Never commit .env files
```

---

## platform/bff/AGENTS.md (the override)

```markdown
# AGENTS.md — platform/bff

> Overrides and extends root AGENTS.md for the BFF package only.
> Root rules still apply unless explicitly overridden here.

## Commands (BFF-specific — override root defaults)

- Test:  yarn workspace @my-org/bff test --runInBand
         ↑ --runInBand required: BFF tests share a session store that races
- Build: yarn workspace @my-org/bff build

## BFF-specific conventions

- All HTTP responses MUST go through ResponseInterceptor — never send raw objects
  Correct:  return this.responseInterceptor.wrap(data)
  Wrong:    return { data, status: 'ok' }

- Cache headers are owned by CacheMiddleware — never set Cache-Control manually

- WebSocket handlers live in gateways/ — not in controllers/

## Folder ownership

- src/adapters/   ← Generated by the schema-sync job; DO NOT TOUCH manually

## Links

- BFF architecture: [link to BFF-specific design doc]
- Session store design: [link]
```

---

## How precedence works

When an agent is working in `platform/bff/src/user.controller.ts`:

1. It reads `platform/bff/AGENTS.md` (nearest — wins)
2. It reads root `AGENTS.md` (parent — fills in what the nested file doesn't cover)
3. Any applicable glob rules (`.mdc` files)
4. Your prompt
5. Base model defaults

The BFF test command overrides the root test command. The global "never use npm" still applies. You only write what's *different* in the nested file.

---

## When to create a nested AGENTS.md

Create one when a subdirectory has:
- A different test command or flags
- Package-specific patterns not applicable to the rest of the repo
- Folder ownership rules that would be confusing at root level

Don't create one just to organize. The root file is the right place for rules that apply everywhere.
